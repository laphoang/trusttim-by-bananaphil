TrustTim — AI customer-care assistant for Hanoi Heart Hospital
Team BananaPhil · Live demo: https://trusttim-by-bananaphil.vercel.app/

For patients and families contacting Hanoi Heart Hospital, TrustTim answers booking, BHYT, and procedure questions directly from the hospital's own official information. When a message indicates a possible cardiac emergency, it stops answering and directs the user to emergency care — so no one receives a chatbot response when they need a hospital.

WHAT WE ARE BUILDING
A grounded, Vietnamese-first RAG assistant embedded on the hospital website. It answers the five highest-volume question clusters — booking, examination/treatment procedures, BHYT coverage and pricing, hospital information, and doctor schedules — strictly from the hospital's curated knowledge base. When no confident source exists, it says "I don't know" and refers the user to the official hotline rather than guessing.

The pipeline is safety-first, in fixed order: (1) normalize the query with a Vietnamese synonym/abbreviation dictionary; (2) an emergency guardrail — a gpt-4.1-mini severity classifier labels each message serious/normal/none, returns fixed escalation or redirect copy, and fails safe on any error; (3) an intent/scope guardrail over the five in-scope intents; (4) hybrid retrieval — dense (vietnamese-embedding over pgvector) plus keyword (Postgres FTS), fused via Reciprocal Rank Fusion; (5) reranking (bge-reranker-v2-m3) with a grounding gate; (6) grounded generation with citations via gpt-4.1-mini; (7) a booking call-to-action when booking is a matched intent.

Requirement coverage: FAQ answering (hybrid RAG over the curated KB); booking integration (CTA + mocked booking service); conversational experience (Vietnamese chat widget with distinct states per response type); grounded/no-hallucination with an "I don't know" fallback and citations; emergency detection and escalation (runs before everything else, fails safe); deployment readiness (Dockerfile, environment-swappable endpoints, live on Vercel). This is a deployed demo: booking is mocked and schedule/pricing data is a dated snapshot — deliberate 48-hour scope decisions.

WHO IT IS FOR
The ~2,500–3,000 outpatients and families visiting the hospital each day — a broad, often elderly, often anxious population needing correct logistics answers without waiting on an overloaded hotline or reception queue. Secondarily, the reception and customer-service (CSKH) staff, freed from repetitive enquiries to handle cases that need a person. And critically, the person whose "question" is in fact a cardiac emergency — the one interaction that must never be treated as a routine FAQ, and the reason the safety check runs first.

WHAT STANDS OUT
Clinical-safety rigor — the emergency guardrail. Before retrieval or any answer, a three-way severity classifier runs. A "serious" verdict returns fixed, doctor-authored escalation copy (call 115 / nearest Emergency Department) and raises a support-case handoff; the model never free-generates in this path. A "normal" symptom gets a fixed "cannot examine, please book" redirect. On any classifier error, the system returns the safety notice — safety never depends on a successful model call. The taxonomy is physician-owned and tuned for recall on the serious class, because at a cardiac hospital a missed emergency is unacceptable. This is the hardest and highest-stakes part of the brief to get right.

Grounded, no-hallucination answering. Hybrid retrieval over the hospital's own knowledge base — not the open web or model memory — reranked and passed through a grounding gate, so an in-scope question with no confident source gets "I don't know," not a guess. Every grounded answer links to the hospital's real source page (internal chunk ids stay in logs, not the user-facing reply), with freshness/is_synthetic caveats on time-boxed data, and structured data like BHYT/pricing renders as a clean table.

Vietnamese-first and multi-intent aware. A Vietnamese embedding model, reranker, and synonym/abbreviation dictionary handle how patients actually phrase things; multi-label intent means a message that both books and asks a question is handled in full.

Built to run inside a Vietnamese hospital. Retrieval — the most sensitive operation, vector similarity search over private patient FAQs — runs on FPT AI Factory (Vietnam/Japan data centers), keeping it within Vietnamese-controlled infrastructure; generation and classification run on OpenAI (gpt-4.1-mini). No patient PII is stored; the service is stateless; the KB is protected by Supabase row-level security. It is containerized and environment-swappable — moving generation to dedicated FPT capacity or an on-premises GPU is a config change, not a rewrite.

A real workflow, not just chat. A genuine per-facility booking handoff — CS1 (92 Trần Hưng Đạo) via the hospital's Zalo Mini App, CS2 (695 Lạc Long Quân) via the website booking page, plus the 1900 1082 hotline — gives the conversation a defined outcome.

WHY THIS WINS
A generic entry wires a language model to FAQ documents. TrustTim is the parts that are hard to copy and matter most at a heart hospital: a physician-owned emergency guardrail that runs first and fails safe, grounded and cited answering that declines to guess, and sovereign-cloud, containerized deployment a hospital could realistically pilot.
